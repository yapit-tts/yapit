# Benchmark Gemini 3.1 Flash Lite vs 3.0 Flash for Extraction

## Intent

We currently use `gemini-3-flash-preview` at `ThinkingLevel.MINIMAL` for document extraction. Gemini 3.1 Flash Lite (`gemini-3.1-flash-lite-preview`) costs exactly half per-token:

| Model | Input/1M | Output/1M |
|-------|----------|-----------|
| 3.0 Flash | $0.50 | $3.00 |
| 3.1 Flash Lite | $0.25 | $1.50 |

The question: can 3.1 Flash Lite at higher thinking levels match 3.0 Flash MINIMAL quality — and what does that actually cost once thinking tokens are factored in?

Thinking tokens are billed as output tokens. So at `high` thinking, 3.1 Flash Lite might generate enough thinking tokens to erode or erase the 50% rate advantage. We need empirical numbers.

## Assumptions

- Google's claim that 3.1 Flash Lite "matches 2.5 Flash across key capability areas" suggests it's a generation behind 3.0 Flash at equivalent thinking. Higher thinking might close this gap for structured extraction tasks specifically.
- The extraction prompt is heavily structured (XML-tagged output format, specific rules). This kind of constrained generation might benefit less from thinking than open-ended reasoning — or more, since following complex instructions is exactly what thinking helps with. Need data.
- 3.1 Flash Lite uses `thinkingLevel` (same as 3.0 Flash): `minimal`, `low`, `medium`, `high`.

## Benchmark Design

### Eval matrix

Run the prompt eval harness across:

| Run | Model | Thinking Level |
|-----|-------|----------------|
| baseline | `gemini-3-flash-preview` | `minimal` (current prod) |
| lite-minimal | `gemini-3.1-flash-lite-preview` | `minimal` |
| lite-low | `gemini-3.1-flash-lite-preview` | `low` |
| lite-medium | `gemini-3.1-flash-lite-preview` | `medium` |
| lite-high | `gemini-3.1-flash-lite-preview` | `high` |

### Harness changes needed

The prompt eval harness (`scripts/prompt_eval/run.py`) currently hardcodes the model and doesn't expose thinking level. Needs:

1. `--model` flag on `Pdf`/`Web` subcommands (default: current prod model)
2. `--thinking-level` flag (default: `minimal`)
3. Both stored in `meta.toml` so runs are identifiable and auto-extend works correctly
4. Token usage (input, output, thinking) persisted per-page in a `tokens.json` alongside `stitched.md` — currently only printed to stdout
5. Cost summary in `meta.toml` using actual token counts + model-specific rates

### What to measure

1. **Quality** — agent comparison (existing `compare.py`) across the corpus. Does lite-medium match baseline? Does lite-high exceed it?
2. **Cost per page** — actual $/page from token counts. Not just rate × estimated tokens, but real thinking token volume at each level.
3. **Latency** — duration_ms per page. Thinking adds latency; is it material?
4. **Thinking token ratio** — what fraction of output tokens are thinking tokens at each level? This determines the actual cost multiplier.

## Result

| Run | Cost (55p) | vs Baseline |
|-----|-----------|-------------|
| 3.0 Flash / minimal | $0.2631 | baseline |
| 3.1 Lite / minimal | $0.1316 | -50% |
| 3.1 Lite / low | $0.2587 | -2% |
| 3.1 Lite / medium | $0.1562 | -41% |
| 3.1 Lite / high | $0.3410 | +30% |

Quality: Lite is a distilled model — slightly worse at vision/layout tasks (sentence splits around images, formatting fidelity). Higher thinking levels add more `<yap-speak>` annotations but don't fix the vision regressions. Improvements and regressions roughly cancel out across all thinking levels. Not worth switching — the cost savings don't come with equivalent quality.

## Considered & Rejected

- **Benchmark 3.0 Flash at higher thinking levels** — not useful since we'd never run 3.0 Flash at higher thinking (already expensive at MINIMAL). The whole point is cost reduction.
